## 1. Executive Summary

### 1.1 一句话定义

> 为领健悦见正畸方案设计师打造**以 AI 为核心驱动的智能病例调度工作台**，将设计师从"被动响应状态机器"升级为"主动时间管理专家"。

### 1.2 问题 → 方案 → 影响

| 维度                  | 现状（Before）                           | 目标（After）                              |
| --------------------- | ---------------------------------------- | ------------------------------------------ |
| **优先级感知**  | 5 个 Tab 割裂，靠直觉排序                | AI 动态评分 + 四色分区，秒级感知紧急度     |
| **信息获取**    | 点进详情页才能知道退回原因               | 行内黄金字段 + AI 摘要，零跳转决策         |
| **异常处理**    | 退回→记录→返回→CAD→按记忆改，5步链路 | 悬停预览 + 行内抽屉，2步完成               |
| **产能**        | ★ 30件/人/天                            | ★ 目标37件/人/天（+23%）                  |
| **质量护栏**    | 无系统约束                               | FTR≥70%，SLA超期率≤8%双护栏              |
| **AI 决策支持** | 无                                       | AI 综合助手面板（规划/问答/洞察/批量操作） |

### 1.3 战略价值

领健 **"SaaS + X（软件+商城+AI+智能制造）"** 战略的核心落脚点是 AI 赋能生产效率。本工作台改版是 AI 能力在制造侧的**首个可量化 ROI 的落地场景**，具备以下战略价值：

- **短期**：同等人力成本下提升产能，消化隐形正畸市场高速增长带来的病例量增量
- **中期**：AI 工作台积累的设计行为数据将反哺自动排牙模型训练，形成数据飞轮
- **长期**：构建"AI 辅助设计师"的能力壁垒，难以被竞品快速复制

## 4. 问题陈述与根因分析

### 4.1 核心 HMW 问题

> **How Might We** 让正畸方案设计师在面对 30+ 并发、多状态病例时，
> 无需切换界面或进入详情页，就能在 30 秒内完成优先级判断并立即启动最紧急任务？

### 4.2 三大核心问题的根因分析

#### 问题一：优先级判断成本高

**根因（Root Cause Analysis）：**

```
表象：设计师不知道该先做哪个
  │
  ├── 直接原因：Tab 架构将任务按状态割裂，无跨状态紧急度对比
  │
  └── 根本原因：系统以"状态"为中心设计（待处理Tab/待修改Tab）
               而非以"设计师的下一步行动"为中心设计

现有 Tab 架构的根本缺陷：
  待处理Tab内：无法区分"SLA还有8小时的首版"与"SLA还有1小时的首版"
  跨Tab比较：无法对比"待修改Tab里的急改"与"待处理Tab里的临期病例"哪个更紧
```

**量化损耗（★假设）：**

- 每次登录：2-5 分钟判断优先级
- 每次任务切换（被中断后重新判断）：1-2 分钟
- 合计日均损耗：约 10 分钟/人

#### 问题二：关键信息获取成本高

**根因：**

```
表象：设计师要进详情页才能判断该病例"是什么情况"
  │
  ├── 直接原因：列表页字段未围绕"决策"设计，缺少异常原因摘要
  │
  └── 根本原因：传统列表以"展示数据"为目标，
               而非以"支持下一步行动决策"为目标
               → 导致设计师必须进入详情页才能获得完整决策信息

无效跳转链路：
  列表 → 点击进入详情 → 查看退回意见（1-3分钟）→ 返回列表（上下文断裂）
                                              ↑
                                   这是纯粹的摩擦成本，不产生任何设计价值
```

**量化损耗（★假设）：**

- 每次无效详情页跳转：1-3 分钟
- 日均无效跳转次数：10-15 次
- 合计日均损耗：约 15-30 分钟/人

#### 问题三：异常流处理链路过长

**根因：**

```
当前退回处理流程（7步）：
  Step 1: 发现病例被退回（Tab切换或等待通知）
  Step 2: 点击进入详情页
  Step 3: 阅读医生/Leader的退回意见（可能是长篇大论）
  Step 4: 手动提取关键修改要求（脑力劳动）
  Step 5: 返回列表（上下文断裂）
  Step 6: 打开 CAD 软件
  Step 7: 根据记忆中的修改要求进行修改（高误改风险）

问题节点：
  - Step 3：意见理解成本高（医生表达不规范，如"把门牙收一下"）
  - Step 4：人工提取结构化信息，易遗漏
  - Step 5-7：信息断层，依赖工作记忆，误改率高★约8%
```

**量化损耗（★假设）：**

- 每次退回处理：额外损耗 5-8 分钟（vs 直接在抽屉处理）
- 日均退回处理次数：3-8 次
- 合计日均损耗：约 15-40 分钟/人
- 误改导致二次退回：约 8% 的退回病例需要再次修改

### 4.3 综合损耗估算（★假设）

| 损耗来源       | 日均损耗/人             | 月均损耗/人（22工作日）  |
| -------------- | ----------------------- | ------------------------ |
| 优先级判断     | 10分钟                  | 220分钟（3.7小时）       |
| 无效详情跳转   | 20分钟                  | 440分钟（7.3小时）       |
| 异常流处理冗余 | 20分钟                  | 440分钟（7.3小时）       |
| **合计** | **~50分钟/人/天** | **约18小时/人/月** |

> 假设设计师团队规模200人★，月度系统性损耗约 **3600 人时** = 约 **450 人天**

## 5. 解决方案详细规格

### 5.1 核心设计哲学

**从"状态中心"转变为"行动中心"**

| 旧范式（状态中心）                 | 新范式（行动中心）             |
| ---------------------------------- | ------------------------------ |
| 按状态分 Tab（待处理/待修改/挂起） | 按紧急度分区，所有状态统一列表 |
| 展示病例的当前状态是什么           | 展示"设计师的下一步行动是什么" |
| 设计师主动切换 Tab 寻找任务        | 系统主动将最紧急的任务推到首位 |
| 详情页承载所有信息                 | 列表页承载决策所需的最小信息集 |

**设计约束原则（Design Constraints）：**

1. **零新增认知负担**：新功能不得增加设计师的学习成本，交互必须符合直觉
2. **降级可用性**：每个 AI 功能必须有非 AI 降级方案，确保 AI 不可用时主流程仍可运行
3. **最小信息原则**：列表行字段不超过 8 个，每增加一个字段须有明确的决策价值证明
4. **可信透明度**：AI 评分结果须展示评分依据，避免黑盒造成不信任

### 5.2 Challenge 1：多状态协同与优先级引导

#### 5.2.1 方案：取消 Tab 隔离，引入 AI 动态优先级引擎

**核心机制：综合紧急度评分（SLA Risk Score）**

$$
RiskScore = w_1 \cdot SLA_{remaining} + w_2 \cdot TaskType + w_3 \cdot CollabState + w_4 \cdot DoctorProfile
$$

| 因子                          | 说明                                            | 权重方向 |
| ----------------------------- | ----------------------------------------------- | -------- |
| `SLA_remaining`（剩余时效） | 剩余时间越少，分数越高                          | 负相关   |
| `TaskType`（任务类型权重）  | 退回改稿 > 首版设计（退回更紧迫）               | 正相关   |
| `CollabState`（协作状态）   | 已有医生在线等待确认，分数调高                  | 正相关   |
| `DoctorProfile`（医生画像） | 高频退回医生的病例预留 buffer，适度降低紧迫系数 | 微调     |

> **冷启动策略：** V1.0 上线初期使用纯规则引擎（`w1=0.6, w2=0.3, w3=0.1`），积累 200 例/人 历史数据后逐步引入机器学习权重优化。

**四色优先级分区规则（确定性逻辑）：**

| 分区                       | 颜色 | 触发条件（满足任一）                                                                | 设计师行动指引             |
| -------------------------- | ---- | ----------------------------------------------------------------------------------- | -------------------------- |

| 病例编号             | 唯一标识，便于沟通                       | 标识     | ✅               |
| 优先级色块           | 直接回答紧急度                           | 决策辅助 | ✅               |

#### 5.3.2 列表行布局规格

**视觉层级（F型阅读路径设计）：**

```
← 最重要（左侧，第一眼扫描）              最次要（右侧，按需查看）→

[色块][编号] [患者信息+标签]    [协作信息]   [时间]  [摘要]      [操作]
  主权重      次权重            三权重      四权重  五权重      固定宽度
  60px        200px             160px       100px   200px       120px
```

**各字段详细规格：**

**① 优先级色块 + 病例编号**

- 色块宽度：4px 左边框 + 背景色浅染（红/黄/蓝/灰）
- 红区附加：火焰图标🔥（CSS animation, 2s cycle）
- 病例编号：`#001234`，等宽字体

**② 患者信息 + AI 特征标签 + 优先级理由**

- 主行：患者姓名（14px Bold）+ 年龄（12px Gray）
- 次行（**常显，不再依赖hover触发**）：AI 自动识别的最多 3 个特征标签 + 优先级评分理由标签
  - 特征标签格式：`#拔牙` `#骨性三类` `#强支抗`
  - 特征标签颜色按难度分级：红色 = 高难度，橙色 = 中难度，灰色 = 常规
  - 优先级理由标签（对应 5.2.1 评分透明化要求）：区域色系 + 简短说明，如 `因剩余时效仅85min且医生退回，标记红区`
  - 无法识别时显示：`#待AI分析`（不显示错误）

> **设计决策（ADR更新）：** 原设计中特征标签和摘要在 hover 时才显示（渐进披露第二层）。经原型用户测试，设计师反馈「需要鼠标放上去才能看到关键信息，不够直觉」，决定改为**常显**。代价是行高从 60px 增至 86px（+26px），但信息获取效率提升明显，符合「专业效率工具」定位。

**③ 协作信息**

- 主行：门诊简称（≤8字）
- 次行：医生姓名 + 星级（★★☆，基于历史退回频次）
  - 3星：退回率 < 10%（配合度高）
  - 2星：退回率 10-30%（中等）
  - 1星：退回率 > 30%（高频退回，需预留更多时间）

**④ SLA 倒计时**

- 格式：`剩余 HH:MM`，倒计时动态更新
- 颜色状态：
  - 绿色：> 24 小时
  - 橙色：4-24 小时
  - 红色闪烁：< 4 小时
  - 灰色：已挂起（显示"SLA已暂停"）

**⑤ 状态摘要（所有状态病例均显示，常显）**

- **全部病例均展示摘要**（包括黄区首版、蓝区待审），不仅限于退回/挂起
- 格式：`[状态标签] 摘要文字（≤40字）`
  - 红区退回：`[医生退回] 11、21需压低0.5mm，增加附件`
  - 红区驳回：`[内审驳回] 前牙覆合过深，建议增加咬合垫设计`
  - 黄区首版：`[首版] 局部矫治·非拔牙方案，SLA剩余18.5h`（含关键设计提示，非千篇一律的「新病例」）

- **悬停气泡覆盖范围扩展**：黄区和蓝区病例的摘要区域同样支持 Hover Card
  - 黄区 Hover Card（新增 `new` 类型）：展示「📋 设计要点」，包含核查清单、SLA剩余、医生退回风险提示
  - 蓝区 Hover Card：展示待确认项目清单 + 一键催办按钮

**⑥ 动态操作按钮**

### 5.5 AI 能力规格

#### 5.5.1 AI 意见提炼（NLP）

**功能描述：** 将医生/Leader 的非结构化退回意见（语音转文字、自由文本）转化为结构化修改清单。

**输入：**

- 医生文字意见（可能包含口语化表达，如"把门牙收一下"）
- 语音转文字结果（如通过悦见 App 录音）

**输出：**

- 结构化 To-Do List（动作 + 牙位 + 量化数值）
- 30字以内摘要（用于列表行展示）
- 置信度评分（用于界面标注是否"AI已提炼"还是"待人工确认"）

**示例转译：**

| 原始意见                               | AI 提炼输出                                               |
| -------------------------------------- | --------------------------------------------------------- |
| "11和21需要再往里收收，大概半个毫米吧" | `内收 11, 21（-0.5mm）`                                 |
| "把16往后移一移，增加一个附件在13上"   | `远中移动 16（约-1.5mm）/ 增加 13 附件`                 |
| "整体感觉上前牙区还不够整齐"           | `[待人工确认] 上前牙区调整（意见模糊，需联系医生确认）` |

**降级方案：** 若 NLP 服务不可用，直接展示原始文本前 100 字 + "查看完整意见"链接。

#### 5.5.2 AI 特征标签提取

**功能描述：** 从病例资料中自动提取影响设计难度的关键特征标签。

**数据来源：** 病例申请单中的医生填写字段（拔牙计划、骨性分类、附件要求等）

**标签体系：**

| 标签类别 | 示例标签                              | 难度颜色  |
| -------- | ------------------------------------- | --------- |
| 拔牙计划 | `#拔牙` `#非拔牙`                 | 红色/灰色 |
| 骨型分类 | `#骨性三类` `#骨性二类` `#I类`  | 红/橙/灰  |
| 特殊需求 | `#强支抗` `#垂直控制` `#深覆合` | 橙色      |
| 矫治类型 | `#全牙弓` `#局部矫治`             | 灰色      |

**降级方案：** 无法自动识别时显示 `#待确认`，不显示错误。

#### 5.5.3 AI 图像预检（CV）

**功能描述：** 在病例进入设计师队列之前，对口腔扫描图像进行质量自动检测。

**检测维度：**

1. **完整性检测：** 是否缺少关键区域（磨牙区、前牙区、咬合面等）
2. **清晰度检测：** 图像分辨率和噪点是否在可用阈值内
3. **文件格式校验：** CBCT、口扫文件格式是否符合系统要求

**处理逻辑：**

```
医生上传资料
    ↓
AI 图像质量检测（< 60秒）
    ↓
    ├── 检测通过 → 病例正常进入设计师队列（黄区）
    ├── 检测警告（建议补充但不阻断）→ 进入队列，列表行显示⚠️提示
    └── 检测失败（关键资料缺失）→ 自动挂起 + 自动发送补传提醒给诊所
```

**质量目标：**

- 召回率（缺失资料能被识别出来）：≥ 95%（上线后评估）
- 误判率（合格资料被误判为缺失）：≤ 2%（上线后评估）

**降级方案：** CV 服务不可用时，病例正常入队，在列表行标注 `[预检待人工核查]`，不阻断流程。

### 5.7 三智能体架构

#### 5.7.1 架构概述：Hub-and-Spoke 拓扑

悦排工作台以**三个专职 AI 智能体**支撑三大核心界面，采用 **Hub-and-Spoke（轮毂辐射）拓扑**：Agent 1 作为中央 Hub，与 Agent 2、Agent 3 双向连接；Agent 2 与 Agent 3 之间**不直接通信**，需要跨域信息时统一向 Agent 1 请求，由 Hub 过滤精简后下发，避免信息污染与职责越界。

**拓扑结构：**

```
                    ┌─────────────────────────────┐
                    │      Agent 1（通用 Hub）      │
                    │  工作台综合信息处理智能体      │
                    │                             │
                    │  • 信息路由与上下文控制       │
                    │  • 任务排序与优先级计算       │
                    │  • 内部资料知识查询（RAG）    │
                    │  • 跨智能体信息聚合与下发     │
                    └──────────────┬──────────────┘
                                   │
                    ┌──────────────┴──────────────┐
                    │         双向连接              │
                    │   （Agent 2/3 互不直连）      │
          ┌─────────┴──────┐            ┌──────────┴─────────┐
          │  Agent 2        │            │  Agent 3            │
          │  沟通协作智能体  │            │  图像识别智能体      │
          │                 │            │                     │
          │ • 医生偏好画像  │            │ • 图纸问题识别       │
          │ • 退回意见转译  │            │ • 修改建议生成       │
          │ • 沟通草稿生成  │            │ • 模板推荐匹配       │
          └────────┬────────┘            └──────────┬──────────┘
                   │                               │
          ┌────────▼────────┐            ┌──────────▼──────────┐
          │   沟通专属 DB    │            │   图像专属 DB         │
          │ • 医生画像库     │            │ • 标准模板图纸库      │
          │ • 对话历史记录   │            │ • 历史分析记录        │
          │ • 偏好模板库     │            │ • 修改建议记录        │
          └─────────────────┘            └─────────────────────┘
                                   │
                          ┌────────▼────────┐
                          │   通用核心 DB    │
                          │ • 病例历史数据   │
                          │ • SLA 记录      │
                          │ • 任务状态数据  │
                          │ • 全局知识库    │
                          └─────────────────┘
```

**核心设计原则：**

| 原则                     | 说明                                                                     |
| ------------------------ | ------------------------------------------------------------------------ |
| **Hub 路由控制**   | Agent 2/3 所有跨域信息请求经 Agent 1 中转，Hub 负责权限判断与上下文裁剪  |
| **专属 DB 自学习** | 每个智能体维护独立数据库，随使用积累领域知识，越用越准                   |
| **信息最小化下发** | Agent 1 不将全量数据转发，仅下发当前任务所需的精简上下文，防止信息过载   |
| **Hub 并发调度**   | Agent 1 作为性能瓶颈点，需支持并行调度（Agent 2/3 同时请求时不串行排队） |

**三个专属数据库的积累价值：**

```
沟通 DB（随时间增长）：
  王医生 → 偏好：参数精确到0.5mm级别，不接受"大概"表述，历史退回率67%
  冯医生 → 偏好：全牙弓方案倾向保守，SLA宽裕时可多次沟通确认

图像 DB（随使用增长）：
  模板 #T-027 → 最常用于骨性三类病例，平均设计时效18min，首次通过率82%
  模板 #T-043 → 磨牙远中移动专用，建议配合强支抗附件使用

通用 DB（实时同步）：
  当前病例状态、SLA 进度、设计师产能数据（为 Agent 1 路由决策提供实时依据）
```

**各智能体能力、底层技术与可行性：**

```
┌──────────────────┬──────────────────┬───────────────────────────┐
│  Agent 1（Hub）  │  Agent 2（沟通）  │  Agent 3（图像）           │
├──────────────────┼──────────────────┼───────────────────────────┤
│ 底层：LLM + RAG  │ 底层：LLM        │ 底层：专业视觉模型 + LLM   │
│ 可行性：★★★★★   │ 可行性：★★★★☆   │ 可行性：★★★☆☆（分层）    │
└──────────────────┴──────────────────┴───────────────────────────┘
```

#### 5.7.2 Agent 1：工作台综合信息处理智能体

**职责范围：** 对应工作台页面，处理所有结构化信息与任务调度类请求。

| 能力                 | 实现方式                                |
| -------------------- | --------------------------------------- |
| 任务排序与优先级计算 | 规则引擎 + LLM 打分，结合 SLA/医生画像  |
| 退回意见 NLP 提炼    | 实体抽取，输出结构化修改清单            |
| 内部资料知识查询     | RAG（病例库/模板库/历史方案向量化）     |
| 今日工作规划生成     | Function Calling + 数据聚合，带时间估算 |
| 实时风险预警         | 监控 SLA 阈值，触发主动推送             |

**技术选型建议：** Claude 3.5 Sonnet / GPT-4o（对话 + 推理），配合向量数据库（病例历史、标准模板）。

**主要落地风险：** 病例管理系统 API 打通，是工程依赖而非 AI 技术风险。

#### 5.7.3 Agent 2：沟通协作智能体

**职责范围：** 对应沟通协作页面，管理设计师与医生、内部 Leader 的协作关系。

| 能力                   | 实现方式                                         |
| ---------------------- | ------------------------------------------------ |
| 医生偏好与历史画像记录 | 对话历史 + 结构化 Memory（退回率/沟通风格/偏好） |
| 退回意见实时转译       | NLP 参数化提取（口语→标准牙位参数）             |
| 预沟通草稿生成         | Few-shot 提示 + 医生画像个性化适配               |
| 沟通摘要与要点提炼     | 多轮对话压缩，输出结构化待办                     |
| 高风险医生预警         | 基于历史退回率动态标注 ★ 级别                   |

**技术选型建议：** LLM（对话生成）+ Per-doctor Profile 存储（每位医生独立 Memory 空间）。

**关键限制：** 智能体价值依赖沟通渠道整合——需将微信/App/平台消息统一接入，这是数据管道工程问题，不是 AI 能力瓶颈。

#### 5.7.4 Agent 3：图像识别与修改建议智能体

**职责范围：** 对应 AI 图像分析页面，处理口腔扫描图、方案图纸的识别与修改建议。

**分层架构设计（关键）：**

```
输入层
├── 2D图像（口扫照、X光、面像照）──→ 专业口腔视觉模型（底层）
├── 3D方案截图/CAD导出图         ──→ 专业正畸视觉模型（底层）
└── 结构化参数文件（CAD数值）    ──→ 直接送入 LLM（无需视觉模型）
                                              ↓
                              LLM 层（解读 + 建议 + 模板匹配）
                              ├── 问题原因分析
                              ├── 修改步骤生成
                              ├── 标准模板推荐（RAG 匹配）
                              └── 制图参考与绘图步骤
```

| 能力                   | 实现方式                                           | 可行性          |
| ---------------------- | -------------------------------------------------- | --------------- |
| 图像质量预检           | CV 模型（完整性/清晰度检测）                       | ✅ 高           |
| 问题区域标注（2D）     | 专业口腔 AI 模型（如对接 TPS 正畸专家系统）        | ⚠️ 需专业模型 |
| 量化偏差识别（毫米级） | 专业正畸算法，通用 VLM**不足以**达到临床精度 | ⚠️ 需专业模型 |
| 修改建议与步骤生成     | LLM（基于识别结果，生成自然语言建议）              | ✅ 高           |
| 标准模板推荐与对比     | RAG（模板库向量化）+ LLM 推理                      | ✅ 高           |
| 结构化参数版本对比     | LLM 直接处理（V1 vs V2 参数差异分析）              | ✅ 高           |

**重要说明：** 通用 VLM（GPT-4o Vision、Claude 3.5 Sonnet）可处理基础图像描述，但「11/21 需再压低 0.5mm」等毫米级临床判断依赖专业正畸算法。建议优先对接领健现有 **TPS 正畸专家系统**作为图像分析底层，LLM 负责结果解读与建议生成，避免重复研发。

#### 5.7.5 分阶段落地建议

| 阶段                              | 内容                                                         | 优先级           |
| --------------------------------- | ------------------------------------------------------------ | ---------------- |
| **P0（V1.0 随工作台上线）** | Agent 1 全部能力（纯文本/结构化数据，无图像）                | 立即可做         |
| **P1（V1.5）**              | Agent 2 核心能力（沟通转译、画像记录）；Agent 1 NLP 意见提炼 | 依赖 NLP 中台    |
| **P2（V2.0）**              | Agent 3 参数对比与模板匹配（结构化输入部分）                 | 依赖 CAD 接口    |
| **P3（V2.5+）**             | Agent 3 图像识别（对接 TPS 或专业视觉模型）                  | 依赖专业模型评估 |

### 5.8 AI 综合助手面板

> **新增功能（原型迭代确定，纳入 V1.0）**
> 基于原型用户测试发现：设计师除了需要任务管理外，高度需要「AI决策支持」——知道今天工作顺序、了解高风险医生、批量处理重复操作。因此将 AI 助手面板从附加功能升级为核心功能，永久嵌入工作台右侧。

#### 5.7.1 设计定位

AI 助手面板是工作台的「AI 副驾驶（Copilot）」，而非简单的聊天窗口。其核心价值在于：

- **主动推送**：不等设计师问，主动呈现今日工作规划和风险预警
- **数据洞察**：将原本需要查报表才能看到的信息实时内嵌
- **批量效率**：AI 自动识别可批量执行的重复操作，一键节省时间

#### 5.7.2 四大 Tab 规格

**Tab 1：今日规划（默认展示）**

| 元素           | 内容                                | 交互                         |
| -------------- | ----------------------------------- | ---------------------------- |
| 今日进度条     | 已完成/总任务 百分比 + 数字         | 实时更新                     |
| AI 工作排班    | 按SLA紧急度排序的带时间戳排班表     | 点击直接跳转对应病例         |
| 预计完成时间   | 按退回件~20min/首版~35min估算      | 动态重算（有病例完成时更新） |
| 高退回风险预警 | ★医生关联病例高亮标注 + 建议预沟通 | —                           |
| SLA 达成率预测 | 今日预计达成率 %                    | —                           |

**Tab 2：智能问答**

- 6个快捷提问 Chip：最紧急病例 / 王医生退回率 / 患者退回原因 / 今日完成预测 / 灰区处理建议 / 诊所风险排名
- 自由文本输入，支持关键词匹配（患者名、医生名、病例编号、区域词）
- AI 回复带 0.7-1.2s 延迟感，真实感更强
- 知识库基于当前工作台实时数据，回答与列表状态一致

**Tab 3：数据洞察**

| 卡片             | 内容                                                              |
| ---------------- | ----------------------------------------------------------------- |
| 待处理分布       | SVG 甜甜圈图（红/黄/蓝/灰占比）+ 横向分区条形图                   |
| 医生退回风险     | 1★/2★/3★ 医生数量分布 + 高风险医生具体预警（退回率、关联病例） |
| SLA 风险评估     | 严重风险(<4h)/中度(4-12h)/充裕(>12h) 件数 + 今日SLA达成率预测     |
| 本周完成趋势     | 折线图（含今日数据点高亮）                                        |
| 今日诊所病例排名 | 各诊所病例数 + 风险标注                                           |

**Tab 4：批量操作**

AI 根据当前工作台状态主动推荐批量动作，无需设计师手动发现：

| 批量操作                     | 触发条件                       | 节省时间估算 |
| ---------------------------- | ------------------------------ | ------------ |
| 一键催办所有灰区病例         | 灰区有未催办病例               | ~3min        |
| 生成黄区下午排班表（发邮件） | 黄区 > 3 件                    | ~5min        |
| 蓝区批量发送确认提醒         | 蓝区病例 > 1 件且超过24h未确认 | ~2min        |
| 高风险医生预沟通草稿         | 有 ★医生关联病例              | ~5min        |
| 生成今日工作报告（PDF）      | 随时可用                       | ~8min        |

每个操作附有 AI 分析依据说明，执行后弹 Toast 确认反馈。

## 6. UX 体验深化设计

> 本章从"减少认知负担、减少操作步骤、减少焦虑感"三个维度，对已有方案做体验层的精细化设计。这些优化不改变核心功能逻辑，但直接决定设计师"用起来顺不顺"。

### 6.1 设计原则总纲

| 原则                   | 说明                                 | 反模式                         |
| ---------------------- | ------------------------------------ | ------------------------------ |
| **零启动摩擦**   | 打开即知道该做什么，无需思考         | 打开后还要扫描列表判断优先级   |
| **一次点击完成** | 最高频操作不超过1步                  | 常用操作需要2-3次点击          |
| **渐进信息披露** | 默认显示最少信息，需要时自然展开     | 所有字段同时可见，信息过载     |
| **操作有反馈**   | 每个完成动作给予即时、轻量的正向反馈 | 完成后毫无反应，不知道是否成功 |
| **专业工具感**   | 支持键盘操作，让熟练用户越用越快     | 只能靠鼠标点击，效率天花板低   |

### 6.2 优化一：今日焦点卡（Today's Focus Card）

**问题：** 设计师打开工作台，即使有四色分区，第一眼仍然是一个列表——还是需要扫描才能找到"现在该做什么"。

**方案：** 在列表顶部固定展示一张 AI 生成的每日简报卡，**0 秒内传达最关键信息**。

```
╔═══════════════════════════════════════════════════════════════╗
║  👋 陈薇，早上好                              06-04  09:02    ║
╠═══════════════════════════════════════════════════════════════╣
║  🔴 立即处理：#001 张三（韶华口腔）  ←  剩余 01:25            ║
╠═══════════════════════════════════════════════════════════════╣
║  今日任务：35件  |  已完成：12件  |  建议顺序：红3→黄12→蓝8  ║
║  💡 上午策略：先清红区（~45min），午后推进黄区                 ║
╚═══════════════════════════════════════════════════════════════╝
```

**交互规格：**

- 卡片高度固定 88px，不随滚动消失（sticky 定位）
- 「立即处理」病例名称可直接点击，展开对应行的 Inline Drawer
- 「建议顺序」文字 hover 时展示气泡：说明 AI 建议顺序的推算逻辑
- 全天清零红区后，卡片自动切换为：`🎉 今日红区已全部完成！当前关注黄区`
- 可折叠（用户偏好持久化），折叠后仅显示一行摘要

**降级方案：** AI 建议策略不可用时，仅展示"今日任务件数 + 当前最紧急病例"，不展示策略文字。

---

### 6.3 优化二：Hover Card 可直接操作（Action-enabled Hover Card）

**问题：** 当前设计中 Hover Card 是纯只读预览，用户还需要再点击一次才能操作。对于"催办"这类 10 秒能完成的轻量操作，多一次点击就是一次摩擦。

**方案：** 将最高频的轻量操作直接内嵌进 Hover Card，悬停即可操作，无需展开抽屉。

**退回病例 Hover Card（升级版）：**

```
╔════════════════════════════════════════╗
║ 📋 退回意见           王医生 · 2h前    ║
╠════════════════════════════════════════╣
║ ☑ 1. 压低 11、21（0.5mm）  ← 可勾选   ║
║ ☐ 2. 远中移动 16、26       ← 可勾选   ║
║ ☐ 3. 增加 13、23 附件      ← 可勾选   ║
║ 进度：1/3 完成                         ║
╠════════════════════════════════════════╣
║  [▶ 立即修改（打开CAD）] [展开详情 →]  ║
╚════════════════════════════════════════╝
```

**挂起病例 Hover Card（升级版）：**

```
╔════════════════════════════════════════╗
║ ⏸ 挂起原因           已等待 2 天       ║
╠════════════════════════════════════════╣
║ ✗ 右侧咬合像（必需）                   ║
║ ✗ CBCT（建议提供）                     ║
╠════════════════════════════════════════╣
║  [📨 一键催办]  [📎 上传资料]  [展开→] ║
╚════════════════════════════════════════╝
```

**升级点：**

- 退回意见清单：可在 Hover Card 内直接勾选已完成项，勾选状态同步保存
- 挂起病例：「一键催办」按钮直接在 Hover Card 内可点，点击后按钮变为"已催办 ✓"（2s后消失）
- 保留「展开详情 →」入口，引导需要深度操作的用户进入 Inline Drawer

---

### 6.4 优化三：三层信息渐进披露（Progressive Disclosure）

**问题：** 当列表中所有字段同时可见时，设计师在"扫描"模式下仍有视觉噪音。

**方案：** 根据交互状态，分三层控制信息显示密度。

```
第一层 — 默认（扫描模式）：仅显示决策必需的最少字段
┌──────────────────────────────────────────────────────┐
│ 🔴 #001  张三（24）    剩余 01:25   [立即修改]       │
└──────────────────────────────────────────────────────┘

第二层 — Hover（关注模式）：鼠标移入行，补充出现辅助字段
┌──────────────────────────────────────────────────────┐
│ 🔴 #001  张三（24）  #拔牙 #强支抗  韶华口腔-王医生★★☆ │
│          剩余 01:25  [医生退回] 11、21需压低  [立即修改][⋯]│
└──────────────────────────────────────────────────────┘

第三层 — 展开（操作模式）：点击后 Inline Drawer 展开，完整信息
┌──────────────────────────────────────────────────────┐
│ 🔴 #001 ... （行头信息保持可见）                      │
│ ├── AI 修改清单（左）  |  V1 vs 本次对比（右）        │
│ ├── 医生原文（折叠可展开）                            │
│ └── [导入CAD] [上传V2] [内部沟通] [关闭↑]           │
└──────────────────────────────────────────────────────┘
```

**实现要点：**

- 第一层 → 第二层：CSS transition，100ms，字段淡入（opacity 0→1）
- 字段宽度预留：第一层布局预留第二层字段空间，防止 hover 时行高跳动
- 灰区病例默认也只显示第一层，视觉上更"退后"，减少对红黄区的干扰

---

### 6.5 优化四：完成微反馈系统（Micro-feedback）

**问题：** 设计工作是高重复性的，每完成一件病例没有任何感知反馈，长期产生倦怠感和"不知道今天做了多少"的焦虑。

**方案：** 设计四个轻量、不打断工作流的微反馈节点。

| 触发时机       | 反馈形式                                                   | 持续时间     | 设计原则             |
| -------------- | ---------------------------------------------------------- | ------------ | -------------------- |
| 完成一件病例   | 该行短暂闪绿（#52C41A, 200ms）+ 顶部已完成计数 +1 滚动动效 | 200ms        | 即时、轻微、不打断   |
| 红区计数归零   | 顶部红区图标变为 ✅ +`今日红区已清零` Toast（2秒消失）   | 2s           | 成就感强化，但不弹窗 |
| 今日进度达 80% | Header 达成率数字变绿 + 轻微上升动效                       | 持续保持绿色 | 可见进度里程碑       |
| 一键催办成功   | 按钮文案变为「已催办 ✓」（橙→绿）并在2秒后复原           | 2s           | 操作确认反馈         |

**设计约束：**

- 所有反馈均不使用弹窗（不需要点击确认）
- 动效时长 ≤ 300ms（不阻断操作节奏）
- 颜色反馈不依赖单一颜色通道（同时配合图标，保证色觉无障碍可访问性）

---

### 6.6 优化五：键盘快捷键体系（Keyboard Shortcuts）

**背景：** 正畸方案设计师日常使用专业 CAD 软件，有极强的键盘操作习惯。纯鼠标操作对他们是效率天花板，快捷键能让熟练用户越用越快。

**完整快捷键体系：**

| 快捷键         | 操作                                      | 适用场景   |
| -------------- | ----------------------------------------- | ---------- |
| `J` / `↓` | 列表向下选择下一行                        | 全局       |
| `K` / `↑` | 列表向上选择上一行                        | 全局       |
| `Enter`      | 展开/收起当前行 Drawer                    | 列表聚焦时 |
| `Esc`        | 收起 Drawer / 关闭 Hover Card             | 全局       |
| `Space`      | 显示当前行 Hover Card（键盘触发悬停效果） | 列表聚焦时 |
| `1`          | 触发主操作按钮（开始设计/立即修改）       | 列表聚焦时 |
| `C`          | 一键催办（仅对蓝区/灰区病例生效）         | 列表聚焦时 |
| `Ctrl + F`   | 聚焦搜索框                                | 全局       |
| `Ctrl + 1~4` | 快速筛选红/黄/蓝/灰区                     | 全局       |
| `?`          | 显示快捷键面板                            | 全局       |

**引导设计：**

- 首次登录时，在列表右下角显示 `⌨ 键盘操作提示`（可关闭），引导发现快捷键
- 操作按钮 hover 时，在 tooltip 中显示对应快捷键：`[开始设计] (1)`
- 侧边导航常驻显示 `? 快捷键` 入口

### 7.7 外部商业化路径（工具 SaaS · AI 服务 · Token 消耗）

> **战略背景：** 悦排工作台在内部验证 ROI 后，具备向外部输出的条件。领健本身是 SaaS 公司，将内部工具产品化是 SaaS+X 战略的自然延伸。外部商业化同时能加速数据飞轮——更多外部客户 → 更多设计行为数据 → AI 模型更准确 → 产品竞争力更强。

#### 7.7.1 收入流全景图

```
悦排外部商业化三层收入结构：

  Layer 1 ── 工具 SaaS 订阅（高客单价，按年续费，周期性现金流）
  │  以订阅制向行业内其他隐形正畸厂商开放工作台使用权，按年计费，随时可取消
  │
  Layer 2 ── AI 能力 API 服务（按调用量，高频，线性增长）
  │  NLP 意见提炼 / CV 图像预检 / AI 优先级评分 开放为付费 API
  │
  Layer 3 ── 模型 Token 消耗（使用量计费，随用随付）
```

#### 7.7.4 Layer 3：模型 Token 消耗计费

**背景：** AI 综合助手面板（问答、今日规划、批量操作草稿）每次交互都产生 LLM 调用成本。当悦排向外部客户输出时，这部分成本需要通过合理的 Token 消耗模型转嫁或覆盖。

**成本结构分析（★）：**

```
AI 助手面板的主要 Token 消耗场景：

  今日规划生成：★~2,000 tokens/次（每日1次）
  智能问答：★~500 tokens/次（日均3次）
  批量操作草稿（如工作报告）：★~3,000 tokens/次（日均0.5次）

单设计师日均 Token 消耗：
  ≈ 2,000 + 500×3 + 3,000×0.5 = ★5,000 tokens/人/天

成本估算（以 Claude claude-sonnet-4-6 为例）：
  Input: ★$3/M tokens，Output: ★$15/M tokens（混合约 $5/M）
  日均成本：5,000 × $5/1,000,000 = $0.025 ≈ ★¥0.18/人/天
  月均成本（22天）：★¥4/人/月

200名内部设计师月度 LLM 成本：★¥800/月（可忽略不计）
```

**Token 计费的三种外部定价模式：**

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **含在席位费里（建议）** | SaaS 定价时将 Token 成本内嵌，对用户透明，无感知 | Standard/Professional 订阅客户，月均用量可预测 |
| **按量超额收费** | 基础席位含每月一定 Token 配额（如 100万 tokens/人），超出部分按 ★¥0.06/千 tokens 计费 | Enterprise 客户，用量差异大 |
| **纯按量付费（API 用户）** | 购买 Token 包，用完续充 | Layer 2 API 用户，无工作台席位 |

**定价建议：**
```
Professional 席位 ★¥5,000/人/年 中内嵌成本核算：

  席位收入：¥5,000/人/年 = ¥417/人/月
  LLM 成本（月）：★¥4/人/月
  LLM 成本占比：< 1%

  → Token 成本对毛利几乎无影响，可安全内嵌在席位费中
  → 若未来 AI 用量大幅增长（如深度问答），设置软上限
    超出后提示「本月 AI 额度已用至 80%，建议升级企业版」
```

- [ ] 检测结果分级：通过/警告（建议补充但不阻断）/失败（关键资料缺失）
- [ ] 检测失败 → 病例自动置为"挂起-待补传" + 自动发送补传提醒给诊所（短信/平台消息）
- [ ] 检测警告 → 病例正常进入队列，列表行显示 ⚠️ 标识
- [ ] AI 预检报告在病例抽屉内可查（具体标注缺失项）
- [ ] 误判率（合格资料被判为缺失）上线后监控，目标 ≤ 2%

### Story 6：AI 综合助手面板（新增）

**As** 正畸方案设计师，
**I want** 工作台右侧常驻一个 AI 助手面板，包含今日工作规划、智能问答、数据洞察和批量操作四个模块，
**So that** 我无需切换其他系统即可获得 AI 决策支持，将高频重复操作（催办、提醒、报告）批量完成。

**验收标准：**

_面板通用_

- [ ] AI 助手面板默认展开（300px），嵌入工作台最右侧列，不遮挡病例列表
- [ ] 可折叠：点击「‹」按钮收起，折叠后显示圆角浮动触发器（贴右边缘）
- [ ] 触发器：领健青渐变背景 + AI 图标 + 竖排「AI 助手」白色文字，hover 有微动效
- [ ] 四个 Tab：今日规划 / 智能问答 / 数据洞察 / 批量操作，默认显示「今日规划」

_Tab 1：今日规划_

- [ ] 顶部进度条：展示今日完成率（已完成件数/总件数），完成病例时实时更新
- [ ] AI 工作排班：按 SLA 紧急度排序，每件附预估时长（退回件~20min，首版~35min），含起始时间
- [ ] ★ 医生关联病例在排班中高亮标注「⚠ 退回率高」
- [ ] 底部展示预计完成时间和 SLA 达成率预测

_Tab 2：智能问答_

- [ ] 6个快捷提问 Chip（最紧急病例 / 高风险医生 / 退回原因 / 完成预测 / 灰区建议 / 诊所排名）
- [ ] 自由文本输入框 + 发送按钮，支持 Enter 发送
- [ ] AI 回复基于当前工作台实时数据，延迟感 0.7-1.2s
- [ ] 「正在分析…」占位状态，避免空白等待感

_Tab 3：数据洞察_

- [ ] 待处理分布 SVG 甜甜圈图（红/黄/蓝/灰占比）
- [ ] 医生退回风险分布（1★/2★/3★ 数量）+ 高风险医生具体预警
- [ ] SLA 风险评估（严重/中度/充裕 件数 + 达成率进度条）
- [ ] 本周完成趋势折线图（含今日数据点）
- [ ] 今日诊所病例排名

_Tab 4：批量操作_

- [ ] AI 根据当前状态主动推荐批量动作（不需要设计师主动发现）
- [ ] 每个操作附 AI 分析依据说明（如「节省约8分钟」「已2天未响应」）
- [ ] 支持：一键催办灰区 / 生成黄区排班表 / 批量发确认提醒 / 预沟通草稿 / 生成工作报告
- [ ] 执行后弹 Toast 确认反馈，操作结果明确

## 11. 后续迭代计划

### 11.1 迭代策略总纲

**迭代哲学：验证优先，数据驱动**

```
每个版本遵循"Build → Measure → Learn"闭环：

  V1.0：建立基础框架 → 采集行为基线 → 验证核心假设（+23%产能）
    ↓（Day 60 评估通过）
  V1.5：引入AI能力 → 测量AI准确率和采纳率 → 优化NLP/评分模型
    ↓（数据验证AI价值）
  V2.0：打通制造闭环 → 测量全链路效率 → 验证数据飞轮效应
    ↓（飞轮启动确认）
  V3.0：平台化 → 扩展用户群 → 探索新商业模式
```

**迭代原则：**
1. **不赌技术成熟度**：每版本 AI 功能均设完善的非 AI 降级方案
2. **以数据杀功能**：若 Day 60 某功能采纳率 <10%，立即启动根因分析，不盲目保留
3. **小增量快验证**：V1.5 之后每 4–6 周一个 Minor 版本，而非半年一个大版本
4. **设计师参与共建**：每个 Minor 版本邀请 2–3 名资深设计师参与灰度测试

### 13.3 风险缓解详细说明

#### 风险 R1：AI 评分冷启动

**缓解：三阶段过渡**

1. **第1-3个月（冷启动期）：** 纯规则引擎（剩余SLA×任务类型权重），AI评分作为参考标注
2. **第3-6个月（学习期）：** AI评分逐步接管（设计师达200例历史数据后启用），界面展示"AI评分中"进度
3. **6个月后（稳定期）：** AI主导，规则引擎作为兜底

**透明化设计：** 每个病例的优先级区域旁显示评分依据标注：

> `🔴 红区 | 因: 剩余1.5小时 + 医生退回`

#### 风险 R2：设计师采纳阻力

**缓解：4步采纳计划**

1. 上线前：邀请2-3名资深设计师参与 Beta 测试，获取真实反馈并纳入迭代
2. 上线时：30分钟操作培训（重点展示优先级感知的效率提升）
3. 上线后第2周：在工作台展示"使用AI视图的设计师 vs Tab视图的设计师"产能对比数据
4. 上线后第60天：若AI视图使用率 < 70%，调研原因并快速迭代

#### 风险 R3：功能降级方案

| AI功能       | 不可用时降级方案                             | 用户感知           |
| ------------ | -------------------------------------------- | ------------------ |
| NLP意见提炼  | 展示原始退回文本前100字 + "查看完整意见"链接 | 体验下降但不阻断   |
| CV图像预检   | 标注"待人工核查"，病例正常入队               | 体验下降但不阻断   |
| AI优先级评分 | 规则引擎评分（不感知）                       | 对用户透明         |
| CAD参数传递  | 一键复制参数到剪贴板                         | 体验下降但功能可用 |

## 15. 附录：设计决策日志（ADR）

### ADR-001：取消Tab隔离 vs 保留Tab作为辅助导航

**决策日期：** 2026-06-04
**状态：** 已决策（双模式过渡）

**背景：** 设计师长期使用Tab，完全取消可能引起采纳阻力；但保留Tab会延续"状态中心"的根本问题。

**选项对比：**

| 选项                 | 优势               | 劣势                               |
| -------------------- | ------------------ | ---------------------------------- |
| 完全取消Tab          | 彻底解决根本问题   | 采纳阻力大，可能造成生产力短期下降 |
| 保留Tab + 新增AI视图 | 平滑过渡，降低风险 | 维护两套视图增加工程成本           |
| 仅保留Tab            | 无采纳风险         | 无法解决核心问题                   |

**决策：** 采用"双模式过渡"，默认展示AI视图，保留Tab视图作为用户可选项。在60天后根据AI视图采纳率决定是否下线Tab视图。

---

### ADR-002：Inline Drawer vs 侧边面板（Side Panel）

**决策日期：** 2026-06-04
**状态：** 已决策（Inline Drawer）

**背景：** 两种交互模式都能避免全页面跳转，但有不同的空间和上下文处理方式。

**选项对比：**

| 选项                      | 优势                                                 | 劣势                   |
| ------------------------- | ---------------------------------------------------- | ---------------------- |
| Inline Drawer（行内展开） | 保持上下文（上下行病例仍可见），符合"不打断扫描"原则 | 屏幕纵向空间被压缩     |
| Side Panel（右侧面板）    | 不影响列表空间，信息展示面积更大                     | 列表被遮挡，破坏上下文 |

**决策：** 选择Inline Drawer。设计师的核心诉求是"不离开列表页"，保持对其他病例的感知比展示面积更重要。对于屏幕纵向空间问题，通过限制抽屉最大高度（300px）+ 内部滚动解决。

---

### ADR-003：今日焦点卡 vs 纯列表排序

**决策日期：** 2026-06-04
**状态：** 已决策（纳入V1.0）

**背景：** 四色分区已经解决了优先级可视化问题，是否还需要额外的焦点卡？会不会造成信息冗余？

**选项对比：**

| 选项           | 优势                                                      | 劣势                                             |
| -------------- | --------------------------------------------------------- | ------------------------------------------------ |
| 仅依靠列表排序 | 简洁，无额外UI元素                                        | 设计师仍需扫描列表才能完成"今天该怎么安排"的规划 |
| 今日焦点卡     | 0秒传达最紧急任务，提供AI策略建议，有情绪价值（早安问候） | 占用列表顶部空间，对熟练用户可能冗余             |

**决策：** 纳入，但设计为可折叠。核心理由：新用户和中等熟练度用户每天花在"规划今日工作"上的认知成本是真实的。焦点卡将其从"扫描列表后自行判断"升级为"系统直接告知"，与"行动中心"设计哲学一致。对于不需要的熟练用户，折叠后不影响其工作流。
